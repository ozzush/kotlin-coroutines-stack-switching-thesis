\section{Correctness}
\label{sec:testing}

Kotlin codebase already contains extensive tests for coroutines. We can separate them into
two levels.

\subsection*{Intrinsic semantics tests}

These tests verify the behavior of the lowest-level coroutine
primitives like \startCoroutineUninterceptedOrReturn and \suspendCoroutineUninterceptedOrReturn.
They also cover mid-level coroutine APIs built directly on top of them like \texttt{startCoroutine}
and \texttt{suspendCoroutine}.

The tests are located in the Kotlin repository at
\btexttt{compiler/testData/codegen/box/coroutines/intrinsicSemantics}.

You can find an example of what kind of tests are in the \texttt{intrinsicSemantics} folder in Listing~\ref{lst:intrinsic-semantics-example} in the Appendix~\ref{ch:appendix}).

All discussed Stack-switching implementations pass all intrinsic semantics tests.

\subsection*{Box tests for coroutines}

This is an extensive collection of tests that verifies that coroutines work correctly when used
together with other Kotlin features.
Some of the things that are tested are:
\begin{itemize}
    \item \textbf{Control flow} (tests in \texttt{controlFlow/} directory): testing coroutines
    with \texttt{if}, \texttt{while}, \texttt{try-catch-finally}, and other control flow structures.
    \item \textbf{Default and variadic parameters} (e.g., \btexttt{defaultParametersInSuspend.kt},
    \btexttt{varargCallFromSuspend.kt}): testing suspend functions with default or variadic arguments.
    \item \textbf{Exception handling} (e.g., \texttt{handleException.kt}, \texttt{simpleException.kt}): testing
    that exceptions thrown inside coroutines are correctly passed to the completion continuation or
    handled by \texttt{try-catch} blocks.
\end{itemize}

As an example, below is a short test that verifies coroutine behavior with an \texttt{if} expression:

\begin{lstlisting}[language=Kotlin, caption={Example from controlFlow\_if1.kt}]
suspend fun s1(): Int = suspendCoroutineUninterceptedOrReturn { x ->
    x.resume(42)
    COROUTINE_SUSPENDED
}

fun f1(): Int = 117
fun f2(): Int = 1
fun f3(x: Int, y: Int): Int = x + y

fun box(): String {
    var result = 0
    suspend {
        result = f3(if (f1() > 100) s1() else f2(), 42)
    }.startCoroutine(object : Continuation<Unit> {
        override val context = EmptyCoroutineContext
        override fun resumeWith(result: Result<Unit>) {}
    })

    if (result != 84) return "fail: $result"
    return "OK"
}
\end{lstlisting}

The tests are located in the Kotlin repository at \btexttt{compiler/testData/codegen/box/coroutines}.

Both the initial and the optimized implementations pass 517 out of 523 coroutine tests. The 6 last tests are failing
due to some objects missing casts to \btexttt{Any?} in the generated Wasm code, which is a minor issue that doesn't
interfere with further analysis of the implementations.

\section{Size benchmark}
\label{sec:size-bench}

To evaluate the impact of the Stack-switching implementation on binary size, a small Kotlin project generated by
the Kotlin Multiplatform Wizard~\cite{kotlin-multiplatform-wizard}
was compiled using three different compiler versions:

\begin{itemize}
    \item \textbf{Baseline}: Kotlin 2.3.10 (original implementation without Stack-switching)
    \item \textbf{Stack-switching}: Base Stack-switching implementation rebased to Kotlin 2.3.10
    \item \textbf{Optimized SS}: Stack-switching with immediate resume optimization rebased to Kotlin 2.3.10
\end{itemize}

For each compiler version, two distributions were generated by running the following Gradle tasks:
\begin{itemize}

\section{Performance benchmarks}
\label{sec:performance-bench}

The performance of the implementation was measured by running benchmarks on low-level coroutine primitives.
The following cases are measured:

\begin{itemize}
    \item \btexttt{chainSuspensions.kt}: 20 coroutines are created and suspended. When the first coroutine is resumed, it
    suspends again, and in the suspend lambda the second coroutine is resumed. The second coroutine suspends and
    resumes the third coroutine in its suspend lambda. This continues on until the last coroutine is resumed, which
    records the fact that it was reached in a global variable. After that, all coroutines that were suspended so far
    are resumed so that they terminate properly.
    \item \btexttt{deepSuspensions.kt}: same setup as in \btexttt{chainSuspensions.kt}, but when the last coroutine
    in the chain is resumed, it also suspends and resumes 20 times.
    \item \btexttt{multipleSuspensions1.kt}: create one coroutine and suspend and resume it 50 times. A pattern
    with immediate resume inside suspend lambda is used.
    \item \btexttt{multipleSuspensions2.kt}: same as \btexttt{multipleSuspensions1.kt}, but the immediate resume pattern
    is not used. For each suspension the continuation is saved and resumed from the main context.
    \item \texttt{sequence.kt}: this benchmark is an exception as it's not a benchmark on low-level primitives. It's a
    benchmark for a Kotlin sequence builder, which is built on top of coroutines. For this test a sequence builder
    is used to compute the first 20 numbers from the Fibonacci sequence.
    \item \btexttt{startCoroutinePerformance.kt}: start 100 coroutines with \texttt{startCoroutine}.
\end{itemize}

Four code versions were used for comparison:

\begin{itemize}
    \item \textbf{Baseline} (commit \texttt{359b8303}): The original implementation without Stack-switching
    \item \textbf{Stack-switching} (commit \texttt{96b5d449}): Initial Stack-switching implementation
    \item \textbf{Optimized SS} (commit \texttt{4ffd6205}): Stack-switching with immediate resume optimization
    \item \textbf{Opt + param} (commit \texttt{2f6a6ebb}): Optimized SS with \btexttt{--experimental-wasm-growable-stacks} flag passed to V8
\end{itemize}

The benchmarks were executed in two compilation modes:

\begin{itemize}
    \item \textbf{dev mode}: Development build without optimizations
    \item \textbf{dce mode}: Production build with dead code elimination
\end{itemize}

Each benchmark was run for 50 iterations after 50 warmup iterations. Some iterations were removed
as outliers before computing the average: any measurement exceeding $Q_3 + 1.5 \times IQR$ (where
$Q_3$ is the third quartile and $IQR$ is the interquartile range) was excluded, as such spikes are
typically caused by garbage collection pauses rather than the code under test. The bar charts in this
chapter display the average value across the remaining iterations, with error bars showing the 95\% confidence
interval computed from the filtered sample using the $t$-distribution.

\subsection{Analysis}

One might notice that the new implementation with Stack-switching is comparably slower
than the baseline. While the results for Reference Interpreter are less severe, they do
not represent the experience that the actual users will have.

\btexttt{chainSuspensions} and \btexttt{deepSuspensions} show the worst regression due to the nature of the V8
implementation, showing a 4–4,5 times slow-down. Passing the special parameter,
\btexttt{--experimental-wasm-growable-stacks}, makes the results much better, bringing them down to around 1,5 times
slow-down compared to the baseline, but other cases become slightly worse. Whether the parameter is worth it or not
will depend on a particular application.

Among the presented benchmarks, the one that is the closest in
nature to what one might see in production code is \btexttt{multipleSuspensions2}. This benchmark
is around 2 times slower than the baseline when tested in V8, which is a noticeable regression.

The \btexttt{multipleSuspensions1} benchmark is comparable to the baseline, but the immediate resume pattern used in
this benchmark is very unlikely to be used in real code. This is because resuming a continuation right inside the
suspend lambda contradicts the intention behind suspending in the first place. If the author could compute the resume
value right inside the suspend lambda, they wouldn't need to suspend at all.


\section{Summary}
\label{sec:evaluation-summary}

Stack-switching implementation leads to a modest but noticeable decrease in binary size. The decrease is around 10\%
for a small application with little to no business logic written with Compose Multiplatform. The decrease would not be
as big on an application which has some business logic that doesn't use Kotlin Coroutines, since the part of
the program that doesn't use Kotlin Coroutines is not affected by the new coroutines implementation.

On the other hand,
performance benchmarks show a noticeable regression. The cases that are most similar to what could appear in
real code are around 2 times slower. Cases with intensive stack-switching are 1,5 times slower if an additional V8
parameter is provided and around 4 times slower without it. Cases with the
immediate resume pattern are comparable to the baseline, but they are unlikely to appear in production code.

\section{Note on other VMs}

One might ask why there are no benchmarks for other production VMs. The reason is that other VMs lack some features that are
necessary for the Kotlin Coroutines implementation to function. The following is a list of VMs that were not tested.

\begin{itemize}
    \item \textbf{Wasmer}: according to the Wasm feature roadmap~\cite{wasm-features}, Wasmer fully supports
    Stack-switching. Unfortunately, it doesn't support another Wasm feature that is necessary for Kotlin/Wasm as a
    whole: Garbage collection.
    \item \textbf{SpiderMonkey (Mozilla Firefox)}: Stack-switching is only partially implemented. The implemented
    feature subset is not enough to run the Kotlin Coroutines implementation. While adapting the implementation to run
    only on the supported primitives is technically possible, it is not straightforward and the result would be substantially
    different from the initial implementation~\cite{spidermonkey-stack-switching}.
    \item \textbf{Wasmtime}: Stack-switching implementation in Wasmtime has a bug which prevents Kotlin Coroutines from running properly~\cite{wasmtime-stack-switching-bug}.
\end{itemize}

One more VM that could technically be tested is Wizard~\cite{wizard-engine}. It has support for Stack-switching, but like
Reference Interpreter, it is not a production VM, so the benchmark results would be of limited interest.
