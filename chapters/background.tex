\section{Kotlin Coroutines}
\label{sec:kotlin-coroutines}

\subsection{Introduction}

The Kotlin standard library provides a set of low-level coroutine primitives that define how
coroutines are created, suspended, and resumed. These primitives are the building blocks on top of which
higher-level abstractions are implemented. The \btexttt{kotlinx.coroutines} library, for example,
provides structured concurrency constructs such as \btexttt{launch}, \btexttt{async}, and
\btexttt{Flow} by using these primitives internally.

\subsection{Coroutines example}

On Figure~\ref{fig:kotlin-coroutines-example} you can see an example program built on the low-level
coroutine primitives. It's a simple program that starts a coroutine. After the coroutine suspends, it is resumed
with the value \("\)OK\("\) and the completion saves the result of the coroutine to a variable. The result of
the coroutine is then printed out.

\begin{figure}[H]
    \centering
    \begin{lstlisting}[language=Kotlin, label={lst:kotlin-coroutines-example}]
var continuation: Continuation<String>? = null
var coroutineResult: String? = null

suspend fun suspendFun(): String {
    val result = suspendCoroutineUninterceptedOrReturn { cont ->
        continuation = cont
        COROUTINE_SUSPENDED
    }
    return result
}

object Completion : Continuation<String> {
    override val context: CoroutineContext = EmptyCoroutineContext
    override fun resumeWith(result: Result<String>) {
        coroutineResult = result.getOrThrow()
    }
}

fun main() {
    // returns COROUTINE_SUSPENDED
    ::suspendFun.startCoroutineUninterceptedOrReturn(Completion)
    continuation.resume("OK")
    println(coroutineResult) // prints "OK"
}
    \end{lstlisting}
    \caption{Kotlin coroutines primitives example}
    \label{fig:kotlin-coroutines-example}
\end{figure}

\subsection{Primitives}

Let's have a closer look at the relevant coroutine primitives from the standard library.

\subsection*{\Continuation}

\btexttt{Continuation<T>} represents a coroutine at some point in its execution. That point could be the beginning
of the coroutine if it was not started yet, or some point at which the coroutine was interrupted by calling
\suspendCoroutineUninterceptedOrReturn function. Resuming continuation resumes the coroutine from the recorded point.
Continuation can be understood as \("\)the rest of a coroutine.\("\) The interface has
two main components:

\begin{itemize}
    \item \texttt{context: CoroutineContext} – the context of the coroutine
    \item \texttt{resumeWith(result: Result<T>)} - resumes the coroutine execution, passing a result value that becomes
    the return value of the \suspendCoroutineUninterceptedOrReturn call
\end{itemize}

\subsection*{Suspend functions}
\label{subsec:suspend-funs}

Suspend functions in Kotlin represent a computation that can be suspended and resumed later.

Every suspend function receives an additional parameter \texttt{completion}
of type \Continuation. This completion is resumed after the suspend function
completes. This means that whenever you see that a function has type \texttt{suspend () -> T}, the actual type
of the function after a round of lowerings during compilation will become \texttt{(completion: Continuation<T>) -> T}.

\subsection*{\startCoroutineUninterceptedOrReturn}

This primitive starts a coroutine from the given suspend function:

\begin{lstlisting}[language=Kotlin, basicstyle=\small\ttfamily]
(suspend () -> T).startCoroutineUninterceptedOrReturn(
    completion: Continuation<T>
): Any?
\end{lstlisting}

\startCoroutineUninterceptedOrReturn immediately starts the suspend function with the given \texttt{completion}
continuation as the completion parameter (remember, suspend functions are appended an additional completion parameter
during compilation). The \texttt{completion} parameter is the continuation that will be resumed when the coroutine
finishes its execution (either with a result or an exception).
The function \startCoroutineUninterceptedOrReturn returns:
\begin{itemize}
    \item \COROUTINESUSPENDED if the coroutine suspends
    \item The result value if the coroutine completes immediately without suspension
\end{itemize}

\subsection*{\createCoroutineUnintercepted}

This primitive creates a coroutine without starting it:

\begin{lstlisting}[language=Kotlin, basicstyle=\small\ttfamily]
(suspend () -> T).createCoroutineUnintercepted(
    completion: Continuation<T>
): Continuation<Unit>
\end{lstlisting}

\subsection*{\suspendCoroutineUninterceptedOrReturn}

This primitive gives access to the current continuation inside a suspend function. It allows the function
to control whether it suspends or returns immediately:

\begin{lstlisting}[language=Kotlin, basicstyle=\small\ttfamily]
suspendCoroutineUninterceptedOrReturn<T>(
    block: (Continuation<T>) -> Any?
): T
\end{lstlisting}

The \texttt{block} receives the current continuation and can:
\begin{itemize}
    \item Return \COROUTINESUSPENDED to indicate that the coroutine has suspended and will be resumed later
    \item Return a value without suspension
\end{itemize}

\subsection{Internal behavior}

To understand how these primitives interact with each other internally, see Figure~\ref{fig:coroutine-primitives-full}.
\startCoroutineUninterceptedOrReturn primitive creates a coroutine object \btexttt{CoroutineImpl} from a suspend
function and resumes it. Internally, the \btexttt{CoroutineImpl::resumeWith} method calls the
\btexttt{doResume} method, which contains the actual logic of resuming the underlying suspend function.

At some point in the execution of a suspend function, a \suspendCoroutineUninterceptedOrReturn primitive may be
encountered. This primitive procures a continuation object which represents the rest of the current coroutine and
passes it to the suspend lambda. Suspend lambda is a block of user code, which is meant to save the continuation
somewhere so that it can be resumed later in case the coroutine suspends. Suspend lambda also dictates whether the
coroutine actually suspends. If suspend lambda returns \COROUTINESUSPENDED, the coroutine will suspend; otherwise,
the execution will continue. If the coroutine suspends, it can only be resumed by resuming the continuation object
that was previously passed to the suspend lambda.

\section{WebAssembly and Stack-switching}
\label{sec:webassembly}

\subsection{Intuition}

Stack-switching proposal introduces a continuation type similar to the Kotlin Continuation described in
Section~\ref{sec:kotlin-coroutines} and instructions to work with it.

A good intuition for understanding resuming and suspending in the Stack-switching proposal is that it is similar to
calling a function inside a try-catch block: the \wasmresume instruction (try block) specifies handlers (catch blocks) for
different \textit{suspension tags} (exception types). A function resumed via continuation can suspend with a \wasmsuspend
instruction. Depending on the suspension tag provided with the \wasmsuspend instruction, execution will jump to a
specific suspension handler. That is where differences from throwing an exception appear:
after jumping to the handler, there will be a new continuation object on the stack. Resuming this continuation will
resume the function that has just suspended. This is in contrast to throwing exceptions: throwing
an exception ends the function completely and it can't be resumed.

On Figure~\ref{fig:stack-switching-primitives} you can see the most important Stack-switching primitives and what they
do.

\subsection{Primitives}
\begin{figure}[H]
    \centering
    \input{diagrams/stack-switching-primitives}
    \caption{Stack-switching primitives}
    \label{fig:stack-switching-primitives}
\end{figure}

Below is an introduction for each of the depicted primitives.

\subsection*{\contnew}
\contnew instruction creates a continuation from a function. Continuation has a type that reflects the signature of the
underlying function.

\subsection*{\wasmresume}
\wasmresume instruction resumes a continuation and specifies what to do when the underlying function suspends by installing
suspension handlers for particular suspension tags. As you may recall from the beginning of the section,
continuation objects have a type that matches the underlying function. Since functions can have arguments,
continuations can also expect some values to be passed to fill those arguments. The \wasmresume
instruction expects these values to be on the stack.

\subsection*{\wasmsuspend}
\wasmsuspend instruction suspends the execution with some suspension tag. Suspension tags also have a function-like
type. The input parameters of a suspension tag are values that will be taken by \wasmsuspend instruction from the stack
and passed to the suspension handler. The result type of a suspension tag describes the values that should appear on
the stack of the suspended function after it's resumed again. It also becomes the input type of the continuation
object that will appear on the stack at the handler. Essentially, it is possible to pass values both
from the main context into continuation and from continuation into the main context, and suspension tags
define the types of the passed values.

\begin{figure}[H]
    \centering
        \begin{lstlisting}[language=wasm, label={lst:wasm-example}, basicstyle=\footnotesize\ttfamily]
;; Tag used to coordinate between generator and consumer:
;; The i32 param corresponds to the generated values passed
;; to consumer; no values are passed from consumer to generator.
(tag $generate (param i32))

(func $generator
 (local $i i32)
 (local.set $i (i32.const 100))
 (loop $loop
  ;; Suspend execution, pass current value of $i to consumer.
  (suspend $generate (local.get $i))
  ;; Decrement $i and exit loop once $i reaches 0.
  (local.tee $i (i32.sub (local.get $i) (i32.const 1)))
  (br_if $loop)
 )
)

(func $consumer
 (local $generator_cont (ref $cont_type))
 ;; Create continuation executing function $generator.
 (local.set $generator_cont (cont.new $cont_type (ref.func $generator)))
 (loop $loop
  (block $on_generate (result i32 (ref $cont_type))
   ;; Resume continuation $generator_cont.
   ;; When $generate tag is thrown,
   ;; go to label $on_generate (points to right after this block)
   (resume $cont_type (on $generate $on_generate) (local.get $generator_cont))
   ;; $generator returned: no more data.
   (return)
  )
  ;; Generator suspended, stack now contains [i32 (ref $cont_type)]
  ;; Save continuation to resume it in next iteration
  (local.set $generator_cont)
  ;; Stack now contains the i32 value yielded by $generator
  (call $print)
  ;; Go to the beginning of the loop (br means branch)
  (br $loop)
 )
)
        \end{lstlisting}
    \caption{Stack-switching example in WebAssembly (adapted from an example from the proposal repository~\cite{wasm-stack-switching-example})}
    \label{fig:stack-switching-example}
\end{figure}

Figure~\ref{fig:stack-switching-example}
gives an example code using Stack-switching in Wasm. This is a simple program with generator and consumer.
Consumer creates a continuation from generator and continuously resumes it and prints the received values.
Generator produces values from 100 down to 1 and passes them to the consumer via the suspend instruction.
If the generator is resumed but all the values have already been passed, it returns. Consumer sees that
the generator has finished and also returns.

\subsection{State of the proposal}

As of today, the Stack-switching proposal is at Phase~3 (Implementation Phase) of the WebAssembly
standardization process~\cite{wasm-features}. At this phase the proposal has a complete
specification text and implementations in WebAssembly runtimes. Currently, the most notable implementation
is the one used in V8 (VM for Chrome and Node.js). The proposal is being actively developed with the goal of
eventually adding it to the WebAssembly standard.
