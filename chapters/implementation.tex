\chapter{Implementation}
\label{ch:implementation}

To implement Kotlin coroutines using Wasm stack-switching proposal we have to implement the intrinsics and primitives
discussed in the previous section. Kotlin coroutine primitives can be mapped to the Wasm stack-switching
primitives pretty closely:
\begin{itemize}
    \item \textbf{Continuation} maps to Wasm continuation objects.

    \item \textbf{createCoroutineUnintercepted} maps to \texttt{cont.new}.

    \item \textbf{startCoroutineUninterceptedOrReturn} maps to \texttt{cont.new} followed by \texttt{resume}.

    \item \textbf{suspendCoroutineUninterceptedOrReturn} maps to the \texttt{suspend} instruction with some additional
    logic to execute suspend lambda (the \texttt{block} argument that is passed to \texttt{suspendCoroutineUninterceptedOrReturn}).
    \item \textbf{Continuation.resumeWith} maps to the \texttt{resume} instruction.
\end{itemize}

There is one unfortunate difference in how suspension is handled in Kotlin, which makes it not so obvious how to
implement it in Wasm. In Kotlin the function \texttt{suspendCoroutineUninterceptedOrReturn}
accepts a lambda which either returns some result or suspends the execution. This lambda accepts an argument: current
continuation. Resuming this continuation will resume the function. The intention of providing this argument is
for the suspension lambda to contain code that saves the continuation in some variable. Then, in case
suspension really happens, the function will be resumed by resuming this saved continuation.

For example, consider this simple suspend function:

\begin{lstlisting}[language=Kotlin, basicstyle=\footnotesize\ttfamily]
var savedContinuation: Continuation<String>? = null

suspend fun suspendHere(): String = 
    suspendCoroutineUninterceptedOrReturn { cont ->
        // Save continuation before suspending
        savedContinuation = cont
        // Indicate suspension occurred
        COROUTINE_SUSPENDED
    }
\end{lstlisting}

In this code, the continuation \texttt{cont} is available inside the lambda and can be saved to
\texttt{savedContinuation} before the function actually suspends. Later, calling
\texttt{savedContinuation?.resume("result")} will resume the suspended function.

However, there is a more subtle case that must also be supported: the continuation can be resumed immediately inside
the suspension lambda. This is illustrated by the following example:

\begin{lstlisting}[language=Kotlin, basicstyle=\footnotesize\ttfamily]
suspend fun suspendFun(): String =
    suspendCoroutineUninterceptedOrReturn { cont ->
        cont.resume("OK")      // Resume immediately
        COROUTINE_SUSPENDED    // Still indicate suspension
    }

fun test() {
    var completionResult: String? = null
    val result = suspendFun.startCoroutineUninterceptedOrReturn(
        object : Continuation<String> {
            override val context = EmptyCoroutineContext
            override fun resumeWith(value: Result<String>) {
                completionResult = value.getOrThrow()
            }
        }
    )
    // result == COROUTINE_SUSPENDED
    // completionResult == "OK"
}
\end{lstlisting}

In this case, even though the continuation is resumed immediately inside the lambda, the function still returns
\texttt{COROUTINE\_SUSPENDED} to the caller. The actual result \texttt{"OK"} is delivered to the completion
continuation via its \texttt{resumeWith} method.

Essentially, the problem is that in Kotlin the continuation object has to be available before we even know whether
a function suspends or not. On the contrary, in Wasm we only get the continuation object for the suspended function
after the function actually suspends. Next we will discuss two ways
to solve this problem.

\subsection{Benefits}
\label{subsec:benefits}

This optimization eliminates unnecessary Wasm stack-switching operations in cases where the continuation is resumed
immediately. Since stack-switching operations involve saving and restoring execution state, avoiding them when possible
improves performance. The optimization is beneficial for coroutines that frequently use immediate
resumption patterns, such as those that check cached values or perform fast-path optimizations before potentially
suspending.
